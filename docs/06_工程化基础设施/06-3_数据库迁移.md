# 06-3 数据库迁移

## 1. 概述

数据库迁移使用 Alembic 管理。所有数据库结构变更通过迁移脚本执行，纳入 Git 版本控制。禁止手动修改数据库结构。

## 2. Alembic 初始化

项目初始化时已完成 Alembic 配置：

```
sage/
├── alembic/
│   ├── versions/          # 迁移脚本存放目录
│   ├── env.py             # Alembic 环境配置
│   └── alembic.ini        # Alembic 配置文件
├── app/
│   └── models/            # SQLAlchemy 模型定义
│       ├── agent.py
│       ├── post.py
│       ├── bar.py
│       └── ...
```

## 3. 迁移流程

### 3.1 修改模型

在 app/models/ 中修改或新增 SQLAlchemy 模型类。

### 3.2 生成迁移脚本

```
alembic revision --autogenerate -m "描述性说明"
```

Alembic 自动对比模型定义与当前数据库结构，生成差异迁移脚本。

### 3.3 审查迁移脚本

打开 alembic/versions/ 中新生成的迁移文件，检查 upgrade() 和 downgrade() 函数是否正确。

### 3.4 执行迁移

```
alembic upgrade head
```

### 3.5 回滚（如需）

```
alembic downgrade -1    # 回退一个版本
alembic downgrade <revision_id>  # 回退到指定版本
```

## 4. 迁移脚本示例

```
"""add agent offline status

Revision ID: a1b2c3d4e5f6
Revises: 9f8e7d6c5b4a
Create Date: 2026-05-14 10:00:00
"""
from alembic import op
import sqlalchemy as sa

def upgrade():
    op.add_column('agents', sa.Column('offline_status', sa.String(50)))

def downgrade():
    op.drop_column('agents', 'offline_status')
```

## 5. 注意事项

- 每次迁移必须同时提供 upgrade() 和 downgrade()，确保可逆
- 迁移脚本纳入 Git 版本控制，与其他代码一同提交
- 首次部署时执行 alembic upgrade head 即可应用所有迁移
- 不要在外部（如 pgAdmin）手动修改数据库结构，避免与迁移记录不一致
- 生产环境迁移前先在开发环境测试

## 6. 种子数据

数据库初始化后的种子数据（平陵市地点、预置梗库等）通过独立的初始化脚本 scripts/seed_world.py 写入，不通过 Alembic 迁移管理。
